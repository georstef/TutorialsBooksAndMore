Το ER Diagram δουλεύτηκε σε CA ErWin [http://erwin.com] για αυτό και ο φάκελος 
περιέχει το αρχείο (.ER1) αλλά έγινε και εξαγωγή σε (.PDF) και (.XPS) για μεγαλύτερη ευκολία.

Το backup της βάσης (.ΒΑΚ) και το sql script (.SQL) έγιναν από SQL SERVER 2012 Express

---------------------------------------------------------------------------------

Προσπάθησα να κρατήσω το diagram σε λογικά επίπεδα και να τραβήξω 
μια "κόκκινη γραμμή" (μιας και είναι της μόδας) σε αυτό το σημείο της ανάλυσης,
γιατί θεώρησα ότι ο σκοπός δεν είναι να φτιάξει κάποιος ένα full πρόγραμμα ticketing
χωρίς αναλυτικές προδιαγραφές, αλλά να δείξει ότι κατέχει τις βασικές γνώσεις (and then some)
ανάλύσης και σχεδιασμού ΒΔ.

---------------------------------------------------------------------------------

Μιας και δεν γνωρίζω τους κανόνες ονοματοδοσίας (naming conventions) της Epsilon Net, 
χρησιμοποίησα αυτούς που έχουμε καταλήξει στην τρέχουσα εταιρεία όπου εργάζομαι.

Επεξήγηση:

1. Όλα τα ονόματα των πινάκων προθεματίζονται από 3 χαρακτήρες 
   για να ξεχωρίζουν ανάλογα με το module στο οποίο ανήκουν
   και να μην υπάρχουν αμφιβολίες για το που ανήκουν κάποιοι πίνακες τύπου Status
   (πχ. pay=payroll για πίνακες μισθοδοσίας,
        wrh=warehouse για πίνακες αποθήκης,
    στην περίπτωσή μας tck=ticket για το module καταγραφής tickets)

2. Όλα τα ονόματα των πινάκων είναι στον ενικό
   (πχ. cusCustomer, tckTicket)

3. Όλα τα πεδία προθεματίζονται με το όνομα του πίνακα που ανήκουν 
   (από αυτό εξαιρούνται τα fk πεδία που έρχονται αυτούσια όπως δηλώθηκαν στον αρχικό πίνακα)

4. Σε όλους τους πίνακες το primary key έχει όνομα <πίνακας>ID

5. Όλα τα primary keys είναι της μορφής pk_<πίνακας>_<πεδίο>

6. Όλα τα default είναι της μορφής df_<πίνακας>_<πεδίο>

7. Όλα τα foreign keys είναι της μορφής fk_<τρέχων πίνακας>_<πίνακας προέλευσης>
